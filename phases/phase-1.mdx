## Overview

Phase 1 performs critical pre-upgrade validation and preparation tasks to ensure a successful migration to Windows 11. This phase **does not modify the operating system** but validates readiness and creates safety mechanisms.

<Note>
  Phase 1 is implemented in `roles/fase_1/tasks/main.yml` and runs on all target systems before any upgrade attempt.
</Note>

## Phase 1 Tasks

<Steps>
  <Step title="OS Validation">
    Verify the target system is running Windows 10 Pro
  </Step>
  <Step title="Hardware Compatibility Check">
    Use WhyNotWin11 to validate Windows 11 compatibility requirements
  </Step>
  <Step title="Restore Point Creation">
    Create a system restore point for rollback capability
  </Step>
  <Step title="Windows Update Patching">
    Apply all available Windows updates and reboot if necessary
  </Step>
</Steps>

## 1. OS Validation

### Purpose

Ensures the target system is running **Windows 10 Pro** before attempting any migration steps. Systems running other Windows versions are logged and excluded from further processing.

### Implementation

```yaml
- name: 1. Recolectar Información del PC
  block:
    - name: 1.1 Validacion de Sistema Operativo
      ansible.builtin.assert:
        that:
          - ansible_distribution == "Microsoft Windows 10 Pro"
        success_msg: "El Sistema Operativo SI es Windows 10 Pro"
        fail_msg: "El Sistema Operativo NO es Windows 10 Pro"
      register: so_out

  rescue:
    - name: <ERROR> - 1.1 Recolectar Información del PC
      block:
        - name: <ERROR> - Validacion de SO
          ansible.builtin.debug:
            msg: "No se pudo ejecutar valdiacion de SO, asegúrese de que el PC está ejecutando Windows 10."

        - name: 1.2 Registrar PC No Compatible
          ansible.builtin.copy:
            content: "{{ ansible_fqdn }}: NO es compatible con Windows 10\n"
            dest: roles/fase_1/files/not_compliant_hosts.log
            mode: a
            remote_src: false
          delegate_to: localhost

        - name: 1.3 Falla en ejecucion - Saltar ejecucion de tareas
          ansible.builtin.fail:
            msg: "El Sistema Operativo no es Windows 10. Abordando tareas posteriores"
```

### Error Handling

<Accordion title="What happens when validation fails?">
  1. Debug message is displayed indicating OS validation failure
  2. The hostname is logged to `roles/fase_1/files/not_compliant_hosts.log`
  3. Task execution fails, preventing Phase 2 from running
  4. The system is excluded from the migration batch
</Accordion>

## 2. WhyNotWin11 Compatibility Check

### Purpose

Validates that the hardware and firmware meet all Windows 11 requirements using the **WhyNotWin11** tool. This comprehensive check evaluates 11 different compatibility criteria.

### Setup Process

<Steps>
  <Step title="Create Directory">
    Creates the WhyNotWin11 working directory if it doesn't exist
  </Step>
  <Step title="Copy Executable">
    Deploys WhyNotWin11.exe from the Ansible controller to the target system
  </Step>
  <Step title="Clean Previous Results">
    Removes any existing compatibility report CSV files
  </Step>
  <Step title="Execute Compatibility Check">
    Runs WhyNotWin11 in silent mode and generates a CSV report
  </Step>
</Steps>

### Implementation

```yaml
- name: 2. Verificar la Compatibilidad de Hardware con 'WhyNotWin11'
  when: ansible_distribution == "Microsoft Windows 10 Pro"
  block:
    - name: 2.1 Crear Directorio de 'WhyNotWin11' (si no existe)
      ansible.windows.win_file:
        path: "{{ fase_1_dir_whynotwin11 }}"
        state: directory

    - name: 2.3 Copiar 'WhyNotWin11' a Directorio
      ansible.windows.win_copy:
        src: roles/fase_1/files/WhyNotWin11.exe
        dest: "{{ fase_1_dir_whynotwin11 }}/WhyNotWin11.exe"
        remote_src: false
      when: why_dir_out.stat.exists

    - name: 2.6 Eliminar Reporte (si existe)
      ansible.windows.win_file:
        path: "{{ fase_1_dir_whynotwin11 }}/Salida_Compatibilidad.csv"
        state: absent
      when: compt_out.stat.exists

    - name: 2.7 Ejecutar 'WhyNotWin11' (silencioso) y Generar Archivo CSV
      ansible.windows.win_shell: |
        {{ fase_1_dir_whynotwin11 }}/WhyNotWin11.exe /e CSV {{ fase_1_dir_whynotwin11 }}/Salida_Compatibilidad.csv /s
      args:
        executable: powershell.exe
      register: whynotwin11_output
      when: whynoywin11_stat.stat.exists
```

### Compatibility Criteria Parsing

The CSV output is parsed to extract 11 compatibility factors:

```yaml
- name: 2.9 Analizar la Compatibilidad del PC
  ansible.builtin.set_fact:
    architecture: "{{ csv_content.stdout_lines[1].split(',')[1] }}"
    boot_method: "{{ csv_content.stdout_lines[1].split(',')[2] }}"
    cpu_compatibility: "{{ csv_content.stdout_lines[1].split(',')[3] }}"
    cpu_core_count: "{{ csv_content.stdout_lines[1].split(',')[4] }}"
    cpu_frequency: "{{ csv_content.stdout_lines[1].split(',')[5] }}"
    directx_wddm2: "{{ csv_content.stdout_lines[1].split(',')[6] }}"
    disk_partition_type: "{{ csv_content.stdout_lines[1].split(',')[7] }}"
    ram_installed: "{{ csv_content.stdout_lines[1].split(',')[8] }}"
    secure_boot: "{{ csv_content.stdout_lines[1].split(',')[9] }}"
    storage_available: "{{ csv_content.stdout_lines[1].split(',')[10] }}"
    tmp_version: "{{ csv_content.stdout_lines[1].split(',')[11] }}"
  when: csv_content.stdout_lines | length > 1
```

### Compatibility Requirements

<Tabs>
  <Tab title="Hardware">
    | Requirement | Minimum Specification |
    |-------------|----------------------|
    | **Architecture** | 64-bit CPU and OS |
    | **CPU Compatibility** | Intel Core 8th Gen+ or AMD Ryzen 2000+ |
    | **CPU Cores** | 2 or more |
    | **CPU Frequency** | 1 GHz or higher |
    | **RAM** | 4 GB or more |
    | **Storage** | 64 GB or more on C: drive |
  </Tab>
  <Tab title="Firmware">
    | Requirement | Minimum Specification |
    |-------------|----------------------|
    | **Boot Method** | UEFI (not Legacy BIOS) |
    | **Disk Partition** | GPT (not MBR) |
    | **Secure Boot** | Enabled |
    | **TPM Version** | 2.0 (enabled) |
  </Tab>
  <Tab title="Graphics">
    | Requirement | Minimum Specification |
    |-------------|----------------------|
    | **DirectX** | DirectX 12 or later |
    | **WDDM** | WDDM 2.x driver model |
  </Tab>
</Tabs>

### Compatibility Determination

```yaml
- name: 2.11 Valor Booleano de compatible
  ansible.builtin.set_fact:
    compatible: false

- name: 2.12 Validar si el PC es Compatible con Windows 11
  ansible.builtin.set_fact:
    compatible: true
  when:
    - architecture == 'True'
    - boot_method == 'True'
    - cpu_compatibility == 'True'
    - cpu_core_count == 'True'
    - cpu_frequency == 'True'
    - directx_wddm2 == 'True'
    - disk_partition_type == 'True'
    - ram_installed == 'True'
    - secure_boot == 'True'
    - storage_available == 'True'
    - tpm_version == 'True'
```

<Warning>
  **All 11 criteria must pass** for the system to be marked as `compatible: true`. If any single check fails, the system will not proceed to Phase 2.
</Warning>

```yaml
      Reporte de Compatibilidad para: {{ ansible_fqdn }}

      - Boot Method: {{ boot_method }}
      - CPU Compatibility: {{ cpu_compatibility }}
      - CPU Core Count: {{ cpu_core_count }}
      - CPU Frequency: {{ cpu_frequency }}
      - DirectX + WDDM2: {{ directx_wddm2 }}
      - Disk Partition: {{ disk_partition_type }}
      - Ram Installed: {{ ram_installed }}
      - Secure Boot: {{ secure_boot }}
      - Storage Available: {{ storage_available }}
      - TPM Version: {{ tmp_version }}
    dest: roles/fase_1/files/{{ ansible_fqdn }}-compatibility_report.txt

```

Reports are saved to `roles/fase_1/files/` on the Ansible controller for audit and troubleshooting purposes.

## 3. System Restore Point Creation

### Purpose

Creates a Windows System Restore Point before the upgrade, providing a rollback mechanism in case of migration issues.

<Note>
  This task **only runs on compatible systems** that have passed the OS and hardware validation checks.
</Note>

### Implementation

```yaml
- name: 3. Crear punto de restauración
  when:
    - not compatible # -- OJO -- Valor REAL: compatible
    - ansible_distribution == "Microsoft Windows 10 Pro"
  block:
    - name: 3.1 Comprobar el estado de Proteccion del Sistema
      ansible.windows.win_shell: |
        (Get-ComputerRestorePoint).SequenceNumber
      args:
        executable: powershell.exe
      changed_when: false
      register: restore_points

    - name: 3.2 Activar la Proteccion del Sistema (si no esta activo)
      ansible.windows.win_shell: |
        Enable-ComputerRestore -Drive C:
      when: restore_points.stdout == ""
      register: activate_out

    - name: 3.3 Crear Punto de Restauracion
      ansible.windows.win_shell: |
        Checkpoint-Computer -Description "Pre-Windows11-Migration" -RestorePointType "MODIFY_SETTINGS"
      args:
        executable: powershell.exe
      register: create_point
      changed_when: create_point.stdout == ""
```

### Process Flow

<Steps>
  <Step title="Check System Protection">
    Queries existing restore points to determine if System Protection is enabled
  </Step>
  <Step title="Enable Protection">
    If no restore points exist, enables System Protection on the C: drive
  </Step>
  <Step title="Create Restore Point">
    Creates a restore point named "Pre-Windows11-Migration" with type MODIFY_SETTINGS
  </Step>
</Steps>

<Accordion title="Restore point details">
  - **Description**: Pre-Windows11-Migration
  - **Type**: MODIFY_SETTINGS
  - **Drive**: C: (system drive)
  - **Created**: Before any upgrade tasks execute
  
  Users can manually roll back to this restore point from Windows Recovery Environment if needed.
</Accordion>

## 4. Windows Update Patching

### Purpose

Ensures the Windows 10 system is fully patched before attempting the Windows 11 upgrade. This reduces compatibility issues and ensures the latest security updates are applied.

### Implementation

```yaml
- name: 4. Parches y Reinicio (En caso +)
  when:
    - not compatible # -- OJO -- Valor REAL: compatible
    - ansible_distribution == "Microsoft Windows 10 Pro"
  block:
    - name: 4.1 Validar si hay reinicios pendientes (reiniciar en caso +)
      ansible.windows.win_reboot:
      when: ansible_reboot_pending

    - name: 4.2 Aplicar parches de ser Necesario
      ansible.windows.win_updates:
        category_names: "*"
        reboot: true
      register: update_results
      retries: 3
      delay: 120
      until: update_results.failed_update_count == 0
```

### Update Strategy

<Tabs>
  <Tab title="Pending Reboot Check">
    First checks if a reboot is pending from previous operations and reboots if necessary
    
    ```yaml
    - name: 4.1 Validar si hay reinicios pendientes (reiniciar en caso +)
      ansible.windows.win_reboot:
      when: ansible_reboot_pending
    ```
  </Tab>
  <Tab title="Update Installation">
    Installs all available updates across all categories with automatic reboot
    
    ```yaml
    - name: 4.2 Aplicar parches de ser Necesario
      ansible.windows.win_updates:
        category_names: "*"
        reboot: true
      register: update_results
    ```
  </Tab>
  <Tab title="Retry Logic">
    Retries up to 3 times with 120-second delay if updates fail
    
    ```yaml
      retries: 3
      delay: 120
      until: update_results.failed_update_count == 0
    ```
  </Tab>
</Tabs>

<Warning>
  The patching process may take significant time depending on the number of pending updates. The task will retry automatically if any updates fail to install.
</Warning>

## Phase 1 Completion

After Phase 1 completes successfully:

<Accordion title="Expected outcomes">
  - ✅ OS validated as Windows 10 Pro
  - ✅ Hardware compatibility confirmed (all 11 checks passed)
  - ✅ Compatibility report generated
  - ✅ System restore point created
  - ✅ All Windows updates installed
  - ✅ System rebooted if necessary
  - ✅ `compatible` variable set to `true`
</Accordion>

Systems that complete Phase 1 successfully proceed to **Phase 2: Upgrade**.

## Overview

Phase 2 executes the actual Windows 11 upgrade on systems that have passed Phase 1 validation. The tool supports **two upgrade methods**, each with different characteristics and use cases.

<Note>
  Phase 2 **only executes** when the system is compatible (`compatible: true`) and running Windows 10 Pro.
</Note>

## Upgrade Methods

<CardGroup cols={2}>
  <Card title="ISO-Based Upgrade" icon="disc-drive">
    Uses a Windows 11 ISO image mounted locally for offline installation
    
    **Source**: `roles/fase_2/tasks/main.yml`
  </Card>
  <Card title="Registry-Based Upgrade" icon="sliders">
    Configures Windows Update via registry to pull Windows 11 from Microsoft
    
    **Source**: `roles/fase_2/tasks/regedit_main.yml`
  </Card>
</CardGroup>

## Method 1: ISO-Based Upgrade

### Overview

The ISO-based method provides a controlled, offline upgrade using a pre-downloaded Windows 11 ISO image. This method is faster and doesn't depend on internet connectivity during installation.

<Note>
  Implementation: `roles/fase_2/tasks/main.yml`
</Note>

### Process Steps

<Steps>
  <Step title="WinRM Configuration">
    Copy WinRM restoration script to target system
  </Step>
  <Step title="ISO Deployment">
    Copy Windows 11 ISO to target system
  </Step>
  <Step title="ISO Mounting">
    Mount the ISO and identify the drive letter
  </Step>
  <Step title="Upgrade Execution">
    Run setup.exe with automated parameters
  </Step>
  <Step title="Post-Upgrade Tasks">
    Reboot, verify installation, and apply updates
  </Step>
</Steps>

### 1. WinRM Configuration Preparation

```yaml
- name: 1. Configurar WinRM con Task
  block:
    - name: 1.1 Crear directorio PS
      ansible.windows.win_file:
        path: "{{ fase_2_dir_whynotwin11 }}/PS"
        state: directory

    - name: 1.2 Validar existencia de Directorio PS
      ansible.windows.win_stat:
        path: "{{ fase_2_dir_whynotwin11 }}/PS"
      register: ps_dir_out

    - name: 1.3 Copiar PS a Directorio
      ansible.windows.win_copy:
        src: roles/fase_2/files/WinRM.ps1
        dest: "{{ fase_2_dir_whynotwin11 }}/PS/WinRM.ps1"
        remote_src: false
      when: ps_dir_out.stat.exists
```

<Accordion title="Why is WinRM configuration critical?">
  The WinRM PowerShell script is copied to `C:/temp/ansible/PS/WinRM.ps1` and referenced by the scheduled task created in the main playbook's `pre_tasks`. This ensures that:
  
  - WinRM is automatically reconfigured after each reboot
  - Ansible can reconnect to the system post-upgrade
  - Multiple reboots during the upgrade don't break automation
</Accordion>

### 2. ISO Deployment and Mounting

```yaml
- name: 2. Proceso de ISO para Windows 11
  block:
    - name: 2.1 Crear directorio ISO
      ansible.windows.win_file:
        path: "{{ fase_2_dir_whynotwin11 }}/ISO"
        state: directory

    - name: 2.3 Copiar ISO a Directorio
      ansible.windows.win_copy:
        src: roles/fase_2/files/Win11_23H2_Spanish_Mexico_x64v2.iso
        dest: "{{ fase_2_dir_whynotwin11 }}/ISO/Win11_23H2_Spanish_Mexico_x64v2.iso"
        remote_src: false
      when: iso_dir_out.stat.exists

    - name: 2.4 Montar ISO Windows 11 en {{ ansible_fqdn }}
      ansible.windows.win_shell: |
        Mount-DiskImage -ImagePath "{{ fase_2_dir_whynotwin11 }}/ISO/Win11_23H2_Spanish_Mexico_x64v2.iso"
      when: iso_dir_out.stat.exists

    - name: 2.5 Obtener la letra de la unidad donde se montó la ISO
      ansible.windows.win_shell: |
        Get-Volume -FileSystemLabel "CCCOMA_X64FRE_ES-MX_DV9" | Select-Object -ExpandProperty DriveLetter
      register: iso_mount
      args:
        executable: powershell.exe
```

<Tabs>
  <Tab title="ISO Details">
    | Property | Value |
    |----------|-------|
    | **Version** | Windows 11 23H2 |
    | **Language** | Spanish (Mexico) |
    | **Architecture** | x64 |
    | **Volume Label** | CCCOMA_X64FRE_ES-MX_DV9 |
  </Tab>
  <Tab title="Deployment Path">
    ```
    Source: roles/fase_2/files/Win11_23H2_Spanish_Mexico_x64v2.iso
    Destination: C:/temp/ansible/ISO/Win11_23H2_Spanish_Mexico_x64v2.iso
    Mount Point: Dynamic (queried via PowerShell)
    ```
  </Tab>
</Tabs>

### 3. Upgrade Execution

```yaml
- name: 3. Proceso de Upgrade por ISO
  block:
    - name: 3.1 Ejecutar el instalador de Windows 11 desde la ISO montada
      ansible.windows.win_shell: >
        Start-Process -FilePath "{{ iso_mount.stdout | trim }}:\setup.exe" -ArgumentList
        '/auto upgrade', '/quiet', '/eula accept', '/showoobe none', '/priority normal', '/noreboot',
        '/compat ignorewarning', '/migratedrivers all', '/copylogs C:\temp\ansible' -Wait
      args:
        executable: powershell.exe
      register: upd_w11

    - name: 3.2 Reiniciar el servidor para completar la instalación de Windows 11
      ansible.windows.win_reboot:
        msg: "El servidor se reiniciará para completar la instalación de Windows 11"
        reboot_timeout: 3600  # Ajuste a 1 hora para manejar reinicios prolongados
      when: upd_w11.rc == 0
```

### Setup.exe Parameters Explained

| Parameter | Purpose |
|-----------|----------|
| `/auto upgrade` | Automated upgrade mode (preserves files and settings) |
| `/quiet` | Silent installation without user interaction |
| `/eula accept` | Automatically accept the End User License Agreement |
| `/showoobe none` | Skip Out-of-Box Experience after upgrade |
| `/priority normal` | Run installation at normal process priority |
| `/noreboot` | Prevent automatic reboot (controlled by Ansible) |
| `/compat ignorewarning` | Ignore compatibility warnings (already validated in Phase 1) |
| `/migratedrivers all` | Migrate all drivers from Windows 10 |
| `/copylogs C:\temp\ansible` | Copy installation logs for troubleshooting |

<Warning>
  The `/compat ignorewarning` parameter is safe because Phase 1 already validated all compatibility requirements using WhyNotWin11.
</Warning>

### 4. Post-Upgrade Verification

```yaml
    - name: 3.5 Recoger Facts Windows 11
      ansible.builtin.setup:
      register: facts_w11

    - name: 3.6 Datos de Sistema Operativo W11
      ansible.builtin.debug:
        msg:
          - "Nombre de PC: {{ facts_w11.ansible_facts.ansible_hostname }}"
          - "Version SO: {{ facts_w11.ansible_facts.ansible_distribution }}"
          - "Arquitectura: {{ facts_w11.ansible_facts.ansible_architecture }}"

    - name: 3.7 Generar reporte de Actualizacion
      ansible.builtin.copy:
        content: |
          Reporte de Actualizacion para: {{ facts_w11.ansible_facts.ansible_hostname }}
          -----------------------------------------------
          - Version SO: {{ facts_w11.ansible_facts.ansible_distribution }}
          - Architecture: {{ facts_w11.ansible_facts.ansible_architecture }}
        dest: roles/fase_2/files/{{ ansible_fqdn }}-upgrade_report.txt
        mode: "0644"

    - name: 3.8 Instalar todas las actualizaciones pendientes en Windows 11
      ansible.windows.win_updates:
        category_names: "*"
        reboot: true
      register: update11_results_all
      retries: 3
      delay: 120
      until: update11_results_all.failed_update_count == 0
```

## Method 2: Registry-Based Upgrade

### Overview

The registry-based method configures Windows Update to offer Windows 11 23H2 as an available upgrade. This method uses Microsoft's official update channels and is ideal for environments with good internet connectivity.

<Note>
  Implementation: `roles/fase_2/tasks/regedit_main.yml`  
  This is the default method specified in the main playbook.
</Note>

### Process Steps

<Steps>
  <Step title="Preserve Current Settings">
    Backup existing registry values for rollback capability
  </Step>
  <Step title="Configure Registry">
    Set Windows Update policies to target Windows 11 23H2
  </Step>
  <Step title="Trigger Updates">
    Search, download, and install Windows 11 via Windows Update
  </Step>
  <Step title="Restore Registry">
    Restore or remove registry modifications after upgrade
  </Step>
</Steps>

### 1. WinRM Configuration (Same as ISO Method)

```yaml
- name: 1. Configurar WinRM con Task
  block:
    - name: 1.1 Crear directorio PS
      ansible.windows.win_file:
        path: "{{ fase_2_dir_whynotwin11 }}/PS"
        state: directory

    - name: 1.3 Copiar PS a Directorio
      ansible.windows.win_copy:
        src: roles/fase_2/files/WinRM.ps1
        dest: "{{ fase_2_dir_whynotwin11 }}/PS/WinRM.ps1"
        remote_src: false
      when: ps_dir_out.stat.exists
```

### 2. Registry Backup and Configuration

```yaml
- name: 2. Proceso de Registros W11 en Regedit
  block:
    - name: 2.1 Guardar estado actual de TargetReleaseVersion
      ansible.windows.win_shell: |
        $key = 'HKLM:\SOFTWARE\Policies\Microsoft\Windows\WindowsUpdate'
        $name = 'TargetReleaseVersion'
        if (Test-Path "$key\$name") {
          Get-ItemProperty -Path $key -Name $name | Select-Object -ExpandProperty $name
        }
      register: target_release_version

    - name: 2.2 Guardar estado actual de TargetReleaseVersionInfo
      ansible.windows.win_shell: |
        $key = 'HKLM:\SOFTWARE\Policies\Microsoft\Windows\WindowsUpdate'
        $name = 'TargetReleaseVersionInfo'
        if (Test-Path "$key\$name") {
          Get-ItemProperty -Path $key -Name $name | Select-Object -ExpandProperty $name
        }
      register: target_release_version_info

    - name: 2.3 Verificar si la clave ProductVersion existe
      ansible.windows.win_shell: |
        $key = 'HKLM:\SOFTWARE\Policies\Microsoft\Windows\WindowsUpdate'
        $name = 'ProductVersion'
        if (Test-Path "$key\$name") {
          Get-ItemProperty -Path $key -Name $name | Select-Object -ExpandProperty $name
        }
      register: product_version
```

<Accordion title="Why backup registry values?">
  Before modifying Windows Update policies, the existing values are captured. This allows the playbook to restore the original configuration after the upgrade completes, ensuring that future update behavior returns to the organization's standard policies.
</Accordion>

### 3. Registry Modification

```yaml
    - name: 2.4 Configurar TargetProductVersion
      ansible.windows.win_regedit:
        path: HKLM:\SOFTWARE\Policies\Microsoft\Windows\WindowsUpdate
        name: ProductVersion
        data: "Windows 11"
        type: string

    - name: 2.5 Configurar clave TargetReleaseVersionInfo para la versión de destino
      ansible.windows.win_regedit:
        path: HKLM:\SOFTWARE\Policies\Microsoft\Windows\WindowsUpdate
        name: TargetReleaseVersionInfo
        data: "23H2"
        type: string

    - name: 2.6 Configurar clave TargetReleaseVersion para forzar la actualización a Windows 11
      ansible.windows.win_regedit:
        path: HKLM:\SOFTWARE\Policies\Microsoft\Windows\WindowsUpdate
        name: TargetReleaseVersion
        data: 1
        type: dword

    - name: 2.7 Reiniciar el servidor
      ansible.windows.win_reboot:
        msg: "Reiniciando el servidor para aplicar cambios de registro"
        reboot_timeout: 600
```

### Registry Keys Explained

<Tabs>
  <Tab title="ProductVersion">
    ```yaml
    Path: HKLM:\SOFTWARE\Policies\Microsoft\Windows\WindowsUpdate
    Name: ProductVersion
    Value: "Windows 11"
    Type: String
    ```
    
    **Purpose**: Tells Windows Update to target Windows 11 instead of Windows 10
  </Tab>
  <Tab title="TargetReleaseVersionInfo">
    ```yaml
    Path: HKLM:\SOFTWARE\Policies\Microsoft\Windows\WindowsUpdate
    Name: TargetReleaseVersionInfo
    Value: "23H2"
    Type: String
    ```
    
    **Purpose**: Specifies the exact Windows 11 version (23H2) to upgrade to
  </Tab>
  <Tab title="TargetReleaseVersion">
    ```yaml
    Path: HKLM:\SOFTWARE\Policies\Microsoft\Windows\WindowsUpdate
    Name: TargetReleaseVersion
    Value: 1
    Type: DWORD
    ```
    
    **Purpose**: Enables the target release version policy (1 = enabled)
  </Tab>
</Tabs>

### 4. Windows Update Execution

```yaml
- name: 3. Actualizar paquetes de Windows 11
  vars:
    ansible_ssh_args: -o ControlMaster=no -o ServerAliveInterval=30
  block:
    - name: 3.1 Hacer una pausa de 30 segundos
      ansible.builtin.pause:
        seconds: 30

    - name: 3.2 Buscar actualizaciones manualmente
      ansible.windows.win_updates:
        category_names: "*"
        state: searched
      register: search_results

    - name: 3.3 Descargar actualizaciones
      ansible.windows.win_updates:
        category_names: "*"
        state: downloaded
      register: download_results
      when: search_results.updates | length > 0

    - name: 3.4 Instalar actualizaciones
      ansible.windows.win_updates:
        category_names: "*"
        state: installed
      register: install_results
      when: download_results.updates | length > 0

    - name: 3.5 Reiniciar el servidor para completar la instalación de actualizaciones
      ansible.windows.win_reboot:
        msg: "El servidor se reiniciará para completar la instalación de actualizaciones"
        reboot_timeout: 3600  # Ajuste a 1 hora para manejar reinicios prolongados
      when: install_results.reboot_required
```

<Steps>
  <Step title="Search Phase">
    Windows Update searches for available updates (now including Windows 11 due to registry changes)
  </Step>
  <Step title="Download Phase">
    Downloads the Windows 11 upgrade package from Microsoft servers
  </Step>
  <Step title="Install Phase">
    Installs the downloaded upgrade package
  </Step>
  <Step title="Reboot">
    Reboots to complete the upgrade process (may involve multiple reboots)
  </Step>
</Steps>

<Warning>
  The registry-based upgrade can take significantly longer than ISO-based, as it downloads the entire Windows 11 installation from Microsoft servers. Ensure adequate bandwidth and time allowance.
</Warning>

### 5. Post-Upgrade Updates

```yaml
    - name: 3.4 Instalar todas las actualizaciones pendientes en Windows 11
      ansible.windows.win_updates:
        category_names: "*"
        reboot: true
      register: update11_results_all
      retries: 3
      delay: 120
      until: update11_results_all.failed_update_count == 0

    - name: 3.5 Reiniciar el servidor para completar la instalación de Windows 11
      ansible.windows.win_reboot:
        msg: "El servidor se reiniciará para completar la instalación de Windows 11"

    - name: 3.6 Recoger Facts Windows 11
      ansible.builtin.setup:
      register: facts_w11
```

### 6. Registry Restoration

```yaml
- name: 4. Restaurar o Eliminar Claves de Registro Según Corresponda
  block:
    - name: 4.1 Restaurar el valor original de ProductVersion
      ansible.windows.win_regedit:
        path: HKLM:\SOFTWARE\Policies\Microsoft\Windows\WindowsUpdate
        name: ProductVersion
        data: "{{ product_version.stdout }}"
        type: string
      when: product_version.stdout is defined

    - name: 4.2 Restaurar el valor original de TargetReleaseVersionInfo
      ansible.windows.win_regedit:
        path: HKLM:\SOFTWARE\Policies\Microsoft\Windows\WindowsUpdate
        name: TargetReleaseVersionInfo
        data: "{{ target_release_version_info.stdout }}"
        type: string
      when: target_release_version_info.stdout is defined

    - name: 4.3 Restaurar el valor original de TargetReleaseVersion
      ansible.windows.win_regedit:
        path: HKLM:\SOFTWARE\Policies\Microsoft\Windows\WindowsUpdate
        name: TargetReleaseVersion
        data: "{{ target_release_version.stdout }}"
        type: dword
      when: target_release_version.stdout is defined

    - name: 4.4 Eliminar la clave WindowsUpdate si no existía antes
      ansible.windows.win_regedit:
        path: HKLM:\SOFTWARE\Policies\Microsoft\Windows\WindowsUpdate
        state: absent
      when: product_version.stdout is not defined and target_release_version_info.stdout is not defined and target_release_version.stdout is not defined
```

<Accordion title="Registry restoration logic">
  - If registry values existed before the upgrade, they are **restored** to their original values
  - If the entire `WindowsUpdate` policy key didn't exist before, it is **deleted** to return the system to default behavior
  - This ensures that the system doesn't remain locked to Windows 11 23H2 and can receive future updates normally
</Accordion>

## Method Comparison

<Tabs>
  <Tab title="ISO-Based">
    ### Advantages
    - Faster (no download during installation)
    - Works offline or with limited bandwidth
    - Consistent installation media
    - Easier to troubleshoot
    
    ### Disadvantages
    - Requires pre-downloading and distributing ISO
    - Larger storage footprint on Ansible controller
    - ISO version may become outdated
    
    ### Best For
    - Air-gapped or restricted networks
    - Large-scale migrations
    - Environments with bandwidth constraints
  </Tab>
  <Tab title="Registry-Based">
    ### Advantages
    - Always gets latest Windows 11 version from Microsoft
    - No ISO management required
    - Uses official Microsoft update channels
    - Smaller playbook footprint
    
    ### Disadvantages
    - Requires internet connectivity during upgrade
    - Slower (downloads during installation)
    - Dependent on Microsoft server availability
    - Variable download times
    
    ### Best For
    - Always-online environments
    - Small to medium deployments
    - Organizations preferring Microsoft update channels
  </Tab>
</Tabs>

## WinRM Preservation Strategy

Both upgrade methods implement a **three-layer strategy** to preserve WinRM connectivity:

<Steps>
  <Step title="Pre-Task Scheduled Task">
    Main playbook creates a scheduled task that runs at boot
  </Step>
  <Step title="Phase 2 Script Deployment">
    WinRM.ps1 script is copied to C:/temp/ansible/PS/ before upgrade starts
  </Step>
  <Step title="Automatic Restoration">
    After each reboot, the scheduled task executes WinRM.ps1 to reconfigure WinRM
  </Step>
</Steps>

```yaml
# From main playbook pre_tasks
pre_tasks:
  - name: Crear Tarea Programada para recuperar Script WinRM
    community.windows.win_scheduled_task:
      name: "WinRM Setup"
      description: "Restaurar Configuración de WinRM"
      actions:
        - path: "powershell.exe"
          arguments: "-NoProfile -ExecutionPolicy Bypass -File C:/temp/ansible/PS/WinRM.ps1"
      triggers:
        - type: boot
      state: present
      enabled: true
      user: "SYSTEM"
      logon_type: service_account

# From Phase 2
- name: 1.3 Copiar PS a Directorio
  ansible.windows.win_copy:
    src: roles/fase_2/files/WinRM.ps1
    dest: "{{ fase_2_dir_whynotwin11 }}/PS/WinRM.ps1"
    remote_src: false
```

<Note>
  Without this WinRM preservation strategy, Ansible would lose connectivity to target systems after the upgrade, requiring manual intervention on each machine.
</Note>
